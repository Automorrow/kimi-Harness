# 实现计划工作流

在动手编码之前，通过结构化的方式分析需求、设计方案、拆分任务，确保实现路径清晰可行。

## 何时使用

- 需要实现一个复杂的新功能
- 需要进行大规模重构或架构调整
- 用户明确要求"先做计划再实现"
- 任务涉及多个文件或模块的协调修改

## 工作流

1. **理解需求**：与用户确认功能需求、验收标准、约束条件和优先级，确保对目标有清晰一致的理解
2. **探索现有代码**：阅读相关的现有代码，了解当前架构、代码风格、依赖关系和可扩展点
3. **设计技术方案**：确定整体技术方案，包括数据结构、接口设计、模块划分、关键算法等
4. **识别风险和依赖**：列出潜在的技术风险、外部依赖、需要协调的模块和可能的阻塞点
5. **拆分任务**：将实现工作拆分为可独立执行的小任务，每个任务应有明确的输入、输出和验收标准
6. **确定执行顺序**：按照依赖关系排列任务顺序，标注关键路径和可并行的任务
7. **输出计划文档**：将以上分析整理为结构化的实现计划，供用户审阅确认后再开始编码

## 注意事项

- 计划的粒度要适中：太粗会遗漏细节，太细会限制实现灵活性
- 优先考虑对现有代码的影响，尽量减少破坏性变更
- 计划中应包含测试策略，明确每个阶段的验证方式
- 如果实现过程中发现计划需要调整，及时更新计划并与用户沟通
- 计划不是一成不变的，允许在实现过程中根据新发现进行迭代调整
