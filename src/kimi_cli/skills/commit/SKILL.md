# Git Commit 工作流

指导用户完成高质量的 git commit，确保提交信息清晰、变更范围合理。

## 何时使用

- 完成功能开发或 bug 修复，需要提交代码
- 需要整理暂存区，将变更拆分为多个逻辑提交
- 不确定 commit message 的格式或内容

## 工作流

1. **查看变更概览**：运行 `git status` 和 `git diff --stat`，了解当前所有未提交的变更
2. **分析变更范围**：阅读 `git diff` 的详细内容，判断变更是否属于同一个逻辑单元
3. **拆分提交**（如需要）：如果变更包含多个不相关的逻辑单元，使用 `git add -p` 交互式暂存，将它们拆分为独立的提交
4. **编写 commit message**：遵循以下格式：
   - 第一行：简明扼要的摘要（不超过 50 个字符），使用祈使句
   - 空一行
   - 正文：详细说明变更的原因、影响和上下文（如需要）
   - 关联信息：关联的 issue 编号、PR 编号等
5. **执行提交**：运行 `git commit -m "message"` 或 `git commit` 进入编辑器
6. **确认提交结果**：运行 `git log -1 --stat` 确认提交内容正确

## 注意事项

- 每个提交应该是一个完整的、可独立回退的逻辑单元
- 不要将格式调整、重构和功能变更混在同一个提交中
- commit message 应说明"为什么"做这个变更，而不是"做了什么"（diff 本身已经说明了做了什么）
- 提交前确保代码可以正常构建和通过测试
- 不要提交敏感信息（密钥、密码、token 等）
