# 代码审查工作流

对代码变更进行系统化的审查，从多个维度评估代码质量并提供改进建议。

## 何时使用

- 需要审查 Pull Request 或 Merge Request
- 完成功能开发后需要进行自审
- 用户要求对某段代码或文件进行质量评估
- 需要检查代码是否符合项目规范和最佳实践

## 工作流

1. **了解变更背景**：阅读 PR 描述、关联的 issue、设计文档，理解变更的目的和上下文
2. **查看变更概览**：通过 `git diff --stat` 了解变更涉及的文件和规模，建立整体印象
3. **逐文件审查**：按逻辑顺序阅读每个文件的变更，重点关注：
   - **正确性**：逻辑是否正确，边界条件是否处理，是否有潜在的 bug
   - **可读性**：命名是否清晰，代码结构是否易于理解，注释是否必要且准确
   - **安全性**：是否存在注入风险、敏感信息泄露、权限绕过等安全问题
   - **性能**：是否有明显的性能问题，如 N+1 查询、不必要的循环、内存泄漏等
   - **一致性**：是否符合项目的代码风格和架构约定
4. **检查测试**：确认变更是否有对应的测试覆盖，测试用例是否充分且有意义
5. **汇总审查意见**：将发现的问题按严重程度分类（阻塞、建议、讨论），给出整体评价和改进建议

## 注意事项

- 审查应该提供建设性的意见，说明问题所在和改进方向，而不仅仅是指出问题
- 区分"必须修改"和"建议改进"，尊重作者的设计选择
- 关注变更的整体影响，而不仅仅是代码风格等表面问题
- 如果变更较大，建议拆分为多个小的 PR 以降低审查难度
- 审查时要保持开放心态，自己不熟悉的模式不一定是错误的
