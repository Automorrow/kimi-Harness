# 测试编写工作流

为代码编写高质量的测试，确保功能正确性并提供回归保护。

## 何时使用

- 需要为新功能编写测试
- 需要为已有代码补充测试覆盖
- 采用测试驱动开发（TDD）方式实现功能
- 修复 bug 后需要添加回归测试
- 用户要求检查或提升测试覆盖率

## 工作流

1. **分析测试目标**：明确需要测试的代码单元、功能点和边界条件，梳理输入输出的各种情况
2. **选择测试策略**：根据测试目标确定测试类型（单元测试、集成测试）和测试框架
3. **设计测试用例**：覆盖以下场景：
   - 正常路径：预期的正常输入和输出
   - 边界条件：空值、零值、最大值、最小值等
   - 异常路径：错误输入、异常情况、资源不可用等
   - 典型用例：用户最常遇到的使用场景
4. **编写测试代码**：遵循测试编写原则：
   - 每个测试只验证一个行为点
   - 测试名称清晰描述被测行为（`should_xxx_when_yyy`）
   - 使用合适的断言，确保失败信息有意义
   - 避免测试之间的依赖，每个测试独立运行
5. **运行测试**：执行测试并确保全部通过，检查覆盖率是否达标
6. **审查和完善**：检查测试是否有遗漏的场景，测试代码本身是否清晰可维护

## 注意事项

- 测试应该测试行为而不是实现细节，这样重构时测试不会频繁失效
- 使用 mock/stub 隔离外部依赖，保持测试的确定性和速度
- 避免过度 mock 导致测试失去实际意义
- 测试代码的质量和可读性同样重要，测试也是代码的一部分
- 覆盖率是手段而非目标，追求有意义的覆盖而非数字上的 100%
