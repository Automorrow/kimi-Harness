# 代码简化工作流

识别并简化代码中的复杂度和冗余，提升代码的可读性、可维护性和优雅程度。

## 何时使用

- 代码过于复杂，难以理解和维护
- 存在大量重复代码，需要提取公共逻辑
- 函数或方法过长，需要拆分和简化
- 嵌套层级过深，需要降低复杂度
- 用户明确要求简化或重构某段代码

## 工作流

1. **识别复杂度来源**：阅读目标代码，标注复杂度热点，包括深层嵌套、长函数、重复逻辑、过度抽象、不清晰的命名等
2. **理解业务意图**：在简化之前，先完全理解代码的业务意图和边界条件，确保简化不会改变行为
3. **确认测试覆盖**：确保目标代码有足够的测试覆盖，简化后可以通过测试验证行为一致性
4. **制定简化策略**：根据复杂度类型选择合适的简化手法：
   - 提取函数：将长函数拆分为职责单一的小函数
   - 消除重复：提取公共逻辑为共享函数或工具方法
   - 简化条件：使用早返回（guard clause）减少嵌套，用策略模式替代复杂条件分支
   - 改善命名：使用清晰的变量名和函数名，减少注释需求
   - 移除冗余：删除未使用的代码、不必要的抽象和过度的中间层
5. **逐步实施**：每次只做一种简化，每步都运行测试确认行为不变
6. **验证结果**：运行全部测试，对比简化前后的代码，确认行为一致且可读性提升

## 注意事项

- 简化的前提是"不改变行为"，所有简化都必须通过测试验证
- 不要过度简化导致代码丧失必要的灵活性
- 简化应该提升可读性，而不是追求"聪明"的写法
- 每次只做一种简化，便于回退和审查
- 如果没有测试覆盖，先补充测试再进行简化
