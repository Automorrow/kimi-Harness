# 调试工作流

系统化地定位和修复代码中的 bug，从现象出发逐步缩小范围直到找到根因。

## 何时使用

- 代码运行报错，需要定位问题原因
- 程序行为与预期不符，需要排查逻辑问题
- 出现间歇性错误或难以复现的问题
- 需要分析错误日志或堆栈信息

## 工作流

1. **收集信息**：完整记录错误现象，包括错误信息、堆栈跟踪、输入数据、期望行为与实际行为的差异
2. **复现问题**：确定最小可复现步骤，如果能稳定复现则优先处理；间歇性问题需要收集更多样本
3. **阅读堆栈**：从堆栈跟踪的顶部开始，逐层分析调用链，定位错误发生的具体代码位置
4. **缩小范围**：通过二分法（注释代码、git bisect）、添加日志打印、断点调试等方式逐步缩小嫌疑范围
5. **定位根因**：确认导致问题的根本原因，区分表面症状和深层原因
6. **制定修复方案**：评估修复方案的影响范围，确保不会引入新问题
7. **实施修复**：修改代码，并添加或更新对应的测试用例以覆盖该 bug
8. **验证修复**：运行相关测试，确认 bug 已修复且没有引入回归问题

## 注意事项

- 先理解问题再动手修改，不要盲目尝试
- 优先查看最近的变更（`git log`、`git diff`），很多 bug 是新引入的
- 注意区分"根因"和"症状"，修复根因而不是掩盖症状
- 修复后务必添加回归测试，防止同类问题再次出现
- 如果问题涉及外部依赖（数据库、API、第三方库），先排除外部因素
